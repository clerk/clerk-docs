## What is user enumeration?

**User enumeration** is a technique to determine whether a user account exists on an application, by detecting when failed authentication attempts still return information about the existence or non-existence of user accounts.

For example, if a user attempts to sign in with an identifier that does not match an existing account on your Clerk application, we inform the user that this identifier doesn't match an existing account. This immediate feedback fits the expectations of ordinary users, who may not remember how or whether they have signed up for your application. A malicious actor can take advantage of the fact that the error message discloses whether an account exists for a given identifier (like an email or phone number) to create a list of all of the accounts that exist within an application. This is **user enumeration**.

## Enable user enumeration protection

To remove the ability to determine whether an account exists:

1. In the Clerk Dashboard, navigate to the [**Attack protection**](https://dashboard.clerk.com/~/user-authentication/attack-protection) page.
1. Enable one of the **User enumeration protection** settings based on your needs.

## How user enumeration protection works

### Bulk user enumeration protection

**Bulk user enumeration protection** enforces rate limits to prevent determining which email addresses and phone numbers are registered in bulk, but targeted attacks are still feasible.

Clerk's existing PII protections on our API endpoints are also applied. For example, when signing in, although bulk enumeration protection does reveal if a user exists, Clerk does not reveal their first and last name and redacts most of their phone number ("Send SMS code to +\*\*\*\*\*\*\*\*41").

### Strict user enumeration protection

With **Strict user enumeration protection**, users attempting to sign in or sign up **will no longer receive feedback that reveals if their identifier matches an existing account** until they prove who they are by verifying their identity in some way.

Strict enumeration covers:

- Sign up: If you try to sign up with an email or phone number that already exists, Clerk will not send a verification code. Instead, Clerk will send a notification by email or SMS that someone tried to sign up with an account that exists and you should sign in instead.
- Sign in: If you try to sign in with an email or phone number that already exists, Clerk will respond differently depending on the strategy you use:
  - All password attempts are rejected. Clerk adds artificial latency to defend against hash timing attacks.
  - If you use an email code or email link with an email that does not exist, Clerk will send a notification telling that email that someone tried to sign in but needs to sign up instead.
  - If you use a phone code with a phone that does not exist, Clerk will proceed to the verification screen as if you were sent a message, but will not send anything. (We understand this is a bad user experience for legitimate users, since this looks like bad SMS delivery. But we want to wait to send SMS notification on sign-in until we have CAPTCHA checks and other protections in place.)
- User profile: If you try to add an email or phone number to your user profile that already exists, Clerk will proceed to the verification screen as if you were sent a message, but will not send anything. In this case, Clerk currently does not issue any "good faith" notifications, based on the assumption that the majority of such attempts are likely to be attacks.

### Tradeoffs

With bulk protections enabled, if you try to sign in with an email, but realize you want to switch to a phone code instead, Clerk can give you a helpful reminder of which phone you have on your account. With strict protections enabled, Clerk does not even list phone as an option in our API responses, and the SDK doesn't expose it. (However, if you know your phone number, you could still manually make a request.)
