The `useSSO()` hook is used to create a new SSO flow. It handles the full authentication lifecycle including opening the in-app browser, processing the OAuth callback, and activating the session.

> [!IMPORTANT]
> The redirect URL used by `useSSO()` must be registered in the Clerk Dashboard under **Redirect URLs**. If not registered, the OAuth flow will complete but the session will not be created. See [Redirect URL configuration](#redirect-url-configuration) for details.

## Returns

The `useSSO()` hook returns the `startSSOFlow()` method, which you can use to initiate the SSO flow.

### `startSSOFlow()`

`startSSOFlow()` has the following function signature:

```ts
function startSSOFlow(startSSOFlowParams: StartSSOFlowParams): Promise<StartSSOFlowReturnType>
```

#### Parameters

`startSSOFlow()` accepts the following parameters (`StartSSOFlowParams`):

<Properties>
  - `strategy`
  - [`OAuthStrategy`](/docs/reference/types/sso#o-auth-strategy) | [`EnterpriseSSOStrategy`](/docs/reference/types/sso#enterprise-sso-strategy)

  The strategy to use for authentication. The following strategies are supported:

  - `'oauth_<provider>'`: The user will be authenticated with their [social connection account](/docs/guides/configure/auth-strategies/social-connections/overview). [See a list of supported values for `<provider>`](/docs/reference/types/sso#o-auth-provider).
  - `'enterprise_sso'`: The user will be authenticated either through SAML, EASIE, or OIDC depending on the configuration of their [enterprise SSO account](/docs/guides/configure/auth-strategies/enterprise-connections/overview).

  ---

  - `identifier?`
  - string

  **Required** if `strategy` is set to `'enterprise_sso'`. The [identifier](/docs/guides/configure/auth-strategies/sign-up-sign-in-options) of the user.

  ---

  - `redirectUrl?`
  - `string`

  The full URL or path to redirect to after the SSO flow is complete. If not specified, defaults to the `sso-callback` path using `AuthSession.makeRedirectUri()`. This URL **must** be registered in the Clerk Dashboard under **Redirect URLs**.

  ---

  - `unsafeMetadata?`
  - [`SignUpUnsafeMetadata`](/docs/reference/types/metadata#sign-up-unsafe-metadata)

  Metadata that can be read and set from the frontend and the backend. Once the authentication process is complete, the value of this field will be automatically copied to the created user's unsafe metadata (`User.unsafeMetadata`). One common use case is to collect custom information about the user during the authentication process and store it in this property. Read more about [unsafe metadata](/docs/guides/users/extending#unsafe-metadata).

  ---

  - `authSessionOptions?`
  - `Pick<WebBrowser.AuthSessionOpenOptions, 'showInRecents'>`

  Options passed to `WebBrowser.openAuthSessionAsync()`. Set `showInRecents: true` to keep the browser tab visible in Android's recent apps.
</Properties>

#### Returns

`startSSOFlow()` returns the following:

<Properties>
  - `createdSessionId`
  - `string | null`

  The ID of the session that was created and activated. `null` if the flow was canceled or failed.

  ---

  - `authSessionResult`
  - `WebBrowser.WebBrowserAuthSessionResult | null`

  The result of the web browser session. See the [Expo WebBrowser documentation](https://docs.expo.dev/versions/latest/sdk/webbrowser/#webbrowserauthsessionresult) for more details.
</Properties>

## How to use the `useSSO()` hook

### Basic example

The `useSSO()` hook handles the full flow internally. After `startSSOFlow()` resolves, the session is already active and `useAuth().isSignedIn` will be `true`.

```tsx {{ filename: 'app/(auth)/sign-in.tsx' }}
import { useSSO } from '@clerk/expo'
import { useRouter } from 'expo-router'
import { Text, TouchableOpacity, View } from 'react-native'

export default function SignInScreen() {
  const { startSSOFlow } = useSSO()
  const router = useRouter()

  const handleSSO = async (strategy: 'oauth_google' | 'oauth_apple' | 'oauth_github') => {
    try {
      const { createdSessionId } = await startSSOFlow({ strategy })

      if (createdSessionId) {
        // Session is active — isSignedIn is now true
        // Navigation will happen automatically via your auth layout
      }
    } catch (err) {
      console.error('SSO error:', err)
    }
  }

  return (
    <View>
      <TouchableOpacity onPress={() => handleSSO('oauth_google')}>
        <Text>Continue with Google</Text>
      </TouchableOpacity>
      <TouchableOpacity onPress={() => handleSSO('oauth_apple')}>
        <Text>Continue with Apple</Text>
      </TouchableOpacity>
    </View>
  )
}
```

## Redirect URL configuration

The redirect URL used during the SSO flow **must** be registered in the [Clerk Dashboard](https://dashboard.clerk.com) under **Redirect URLs**. Without this, the OAuth provider will complete authentication but the session will not be created.

### For development builds

If your `app.json` has a `scheme` configured (e.g., `"scheme": "myapp"`), register:

```
myapp://sso-callback
```

### For Expo Go

By default, `useSSO()` generates a redirect URL using `AuthSession.makeRedirectUri({ path: 'sso-callback' })`. In Expo Go, this produces a URL like `exp://192.168.x.x:PORT/--/sso-callback` which changes per network and port. For a stable URL in Expo Go, pass a custom `redirectUrl` using your app's scheme:

```tsx
import * as AuthSession from 'expo-auth-session'

const { createdSessionId } = await startSSOFlow({
  strategy: 'oauth_google',
  redirectUrl: AuthSession.makeRedirectUri({
    scheme: 'myapp',
    path: 'sso-callback',
  }),
})
```

Then register `myapp://sso-callback` in the Dashboard. This works in both Expo Go and development builds.
