---
title: Manage team access
description: Learn about workspace Roles and Permissions in the Clerk Dashboard, including Owner, Admin, Developer, Support, and Viewer Roles.
---

Each workspace in the Clerk Dashboard supports role-based access control to help manage what team members can see and do. When assigning Roles, consider the following:

- Which resources does this team member need access to?
- What actions should this team member be able to perform?
- What level of system configuration access is required?

## Available Roles

The following table summarizes the available Roles:

| Role | Description |
| - | - |
| [**Owner**](#owner) | Full access to all resources, including workspace member management and instance keys. |
| [**Admin**](#admin) | Manage applications, instances, Billing, configuration, users, Organizations, Organization memberships, API keys, and instance keys; can impersonate users. |
| [**Developer**](#developer) | Manage restrictions, view API keys and Billing, and manage users, Organizations, Organization memberships, configuration, and API keys in development instances only; dev-only user impersonation. |
| [**Support**](#support) | Provide user and Organization support with read-only access to application configuration; can manage users, Organizations, Organization memberships, restrictions, and user impersonation. |
| [**Viewer**](#viewer) | Read-only access to users, Organizations, configuration, and workspace-level data; least-privileged Role for basic Dashboard visibility. |

> [!IMPORTANT]
> Only the **Owner** and **Viewer** Roles are available on the Hobby and Pro plans. For the Business plan, all Roles are available.

The Permissions in the tables below are feature-specific. Each table is organized by Dashboard feature or Permission group. The **Existing Permissions** column lists only the Permission types that exist for that feature, and the **Role access** column shows which of those Permissions the Role has. For example, the Users area doesn't have separate create or delete Permissions. [Global](!global) Read grants read access to users, Organizations, and Organization memberships. [Global](!global) Manage grants create, update, and delete access for those records. User impersonation is the user-specific Permission listed separately.

## Owner

The **Owner** Role is the highest level of authority within a workspace, possessing comprehensive access and control over all settings and resources.

**Key responsibilities**

- Oversee and manage all resources and applications within a workspace
- Modify workspace settings, including Billing and member Roles
- Access and modify all applications, including their settings, API keys, and domains
- Manage users, Organizations, Organization memberships, and restrictions _(allowlist, blocklist, waitlist)_
- Impersonate users

**Permissions**

| Feature | Existing Permissions | Role access |
| - | - | - |
| [Global](!global) | Read users, Organizations, and Organization memberships; manage users, Organizations, and Organization memberships (create, update, and delete) | Read and Manage |
| Applications | Read, Manage, Create, and Delete | Read, Manage, Create, and Delete |
| Instances | Read, Manage, Create, and Delete | Read, Manage, Create, and Delete |
| Configuration | Read and Manage | Read and Manage |
| Billing | Read and Manage | Read and Manage |
| Secrets | Manage | Manage |
| Restrictions | Read and Manage | Read and Manage |
| User impersonation | Impersonate users | Impersonate users |

## Admin

The **Admin** Role handles day-to-day management across applications and instances.

**Key responsibilities**

- Manage applications and instances within a workspace
- Modify workspace settings, including Billing and member Roles
- Access and modify all applications, including their settings, API keys, and domains, but **cannot delete or transfer applications**
- Manage users, Organizations, Organization memberships, and restrictions _(allowlist, blocklist, waitlist)_
- Impersonate users

**Permissions**

> [!IMPORTANT]
> Admins **cannot transfer applications** in addition to the Permissions below.

| Feature | Existing Permissions | Role access |
| - | - | - |
| [Global](!global) | Read users, Organizations, and Organization memberships; manage users, Organizations, and Organization memberships (create, update, and delete) | Read and Manage |
| Applications | Read, Manage, Create, and Delete | Read, Manage, and Create |
| Instances | Read, Manage, Create, and Delete | Read, Manage, Create, and Delete |
| Configuration | Read and Manage | Read and Manage |
| Billing | Read and Manage | Read and Manage |
| Secrets | Manage | Manage |
| Restrictions | Read and Manage | Read and Manage |
| User impersonation | Impersonate users | Impersonate users |

## Developer

The **Developer** Role focuses on technical configuration and integrations with limited production access.

**Key responsibilities**

- Manage restrictions _(allowlist, blocklist, waitlist)_
- View API keys and Billing information
- Manage users, Organizations, Organization memberships, configuration, and API keys in development instances only
- Impersonate users in development instances only

**Permissions**

| Feature | Existing Permissions | Role access |
| - | - | - |
| [Global](!global) | Read users, Organizations, and Organization memberships; manage users, Organizations, and Organization memberships (create, update, and delete) | Read; Manage in development instances only |
| Applications | Read, Manage, Create, and Delete | Read |
| Instances | Read, Manage, Create, and Delete | Read |
| Configuration | Read and Manage | Read; Manage in development instances only |
| Billing | Read and Manage | Read |
| Secrets | Manage | Manage in development instances only |
| Restrictions | Read and Manage | Read and Manage |
| User impersonation | Impersonate users | Impersonate users in development instances only |

## Support

The **Support** Role provides tools to assist customers while preventing modifications to sensitive application configurations.

**Key responsibilities**

- Provide direct user and Organization support and troubleshooting
- Manage users, Organizations, and Organization memberships
- Impersonate users for issue resolution and debugging
- Manage restrictions _(allowlist, blocklist, waitlist)_

**Permissions**

| Feature | Existing Permissions | Role access |
| - | - | - |
| [Global](!global) | Read users, Organizations, and Organization memberships; manage users, Organizations, and Organization memberships (create, update, and delete) | Read and Manage |
| Applications | Read, Manage, Create, and Delete | Read |
| Instances | Read, Manage, Create, and Delete | Read |
| Configuration | Read and Manage | Read |
| Billing | Read and Manage | No access |
| Secrets | Manage | No access |
| Restrictions | Read and Manage | Read and Manage |
| User impersonation | Impersonate users | Impersonate users |

## Viewer

The **Viewer** Role has read-only access to users, Organizations, configuration, and workspace-level data.

**Key responsibilities**

- Review configuration settings of applications
- Review users and Organizations
- Review workspace-level information and configuration

**Permissions**

| Feature | Existing Permissions | Role access |
| - | - | - |
| [Global](!global) | Read users, Organizations, and Organization memberships; manage users, Organizations, and Organization memberships (create, update, and delete) | Read |
| Applications | Read, Manage, Create, and Delete | Read |
| Instances | Read, Manage, Create, and Delete | Read |
| Configuration | Read and Manage | Read |
| Billing | Read and Manage | No access |
| Secrets | Manage | No access |
| Restrictions | Read and Manage | No access |
| User impersonation | Impersonate users | No access |
