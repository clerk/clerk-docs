---
title: Manage team access
description: Learn about workspace Roles and Permissions in the Clerk Dashboard, including Owner, Admin, Developer, Support, and Viewer Roles.
---

Each workspace in the Clerk Dashboard supports role-based access control to help manage what team members can see and do. When assigning Roles, consider the following:

- Which resources does this team member need access to?
- What actions should this team member be able to perform?
- What level of system configuration access is required?

## Available Roles

The following table summarizes the available Roles:

| Role | Description |
| - | - |
| [**Owner**](#owner) | Full access to all resources, including workspace member management and instance keys. |
| [**Admin**](#admin) | Manage applications, instances, Billing, configuration, users, Organizations, Organization memberships, API keys, and instance keys; can impersonate users. |
| [**Developer**](#developer) | Manage restrictions, view API keys and Billing, and manage users, Organizations, Organization memberships, configuration, and API keys in development instances only; dev-only user impersonation. |
| [**Support**](#support) | Provide user and Organization support with read-only access to application configuration; can manage users, Organizations, Organization memberships, restrictions, and user impersonation. |
| [**Viewer**](#viewer) | Read-only access to users, Organizations, configuration, and workspace-level data; least-privileged Role for basic Dashboard visibility. |

> [!IMPORTANT]
> Only the **Owner** and **Viewer** Roles are available on the Hobby and Pro plans. For the Business plan, all Roles are available.

The Permissions in the tables below reflect the Dashboard's internal permission groups, not a uniform CRUD model. [Global](!global) Read grants read access to users, Organizations, and Organization memberships. [Global](!global) Manage grants create, update, and delete access for those records. There are no separate Dashboard Permissions such as `users:create` or `users:delete`; user impersonation is the user-specific Permission listed separately.

## Owner

The **Owner** Role is the highest level of authority within a workspace, possessing comprehensive access and control over all settings and resources.

**Key responsibilities**

- Oversee and manage all resources and applications within a workspace
- Modify workspace settings, including Billing and member Roles
- Access and modify all applications, including their settings, API keys, and domains
- Manage users, Organizations, Organization memberships, and restrictions _(allowlist, blocklist, waitlist)_
- Impersonate users

## Admin

The **Admin** Role handles day-to-day management across applications and instances.

**Key responsibilities**

- Manage applications and instances within a workspace
- Modify workspace settings, including Billing and member Roles
- Access and modify all applications, including their settings, API keys, and domains, but **cannot delete or transfer applications**
- Manage users, Organizations, Organization memberships, and restrictions _(allowlist, blocklist, waitlist)_
- Impersonate users

**Permissions**

> Admins **cannot transfer applications** in addition to the Permissions below.

**Key responsibilities**

- Manage restrictions _(allowlist, blocklist, waitlist)_

## Viewer

**Key responsibilities**
